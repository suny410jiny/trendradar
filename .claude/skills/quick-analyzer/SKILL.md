# Quick Analyzer v2.1 (TrendRadar)

작업 요청을 분석하여 복잡도를 판정하고 적절한 플로우로 라우팅한다.

## MCP 활용

- **Sequential-thinking**: 단계별 요청 분석
- **Serena**: 코드 구조 분석, 심볼 검색
- **Postgres (trendradar)**: DB 스키마 확인

---

## 전체 플로우

```
Step 0: 이전 작업 확인
    ↓
Step 1: 복잡도 판정
    ↓
Step 2: 참조 문서 선택
    ↓
Step 3: 계획 수립
    ↓
Step 4: 사용자 승인
    ↓
Step 5: 플로우 라우팅 & 작업 실행
    ↓
Step 6: 문서 동기화
    ↓
실패 시 → 트러블슈팅 → 에스컬레이션
```

---

## Step 1: 복잡도 판정

### Level 0: 즉시 실행 (Direct)
단순 수정, 설정 변경, 1개 파일 수정

**트리거 키워드:**
- 설정 변경, 상수 수정, yml 수정
- 로깅 레벨, 포트, 환경변수

→ 직접 수정

### Level 1: 간소화 플로우 (Light)
단일 기능, 패턴 존재, 2-3개 파일 수정

**트리거 키워드:**
- 필드 추가, 컬럼 추가, 인덱스
- 새 API 엔드포인트 (단순 CRUD)
- 마이그레이션 추가

→ **task-classifier → code-generator** 로 라우팅

### Level 2: 표준 플로우 (Standard)
복합 기능, 분석 필요, 여러 파일 수정

**트리거 키워드:**
- 수집 로직 변경, 스케줄러 수정
- 성능 개선, 쿼리 최적화
- 버그 수정, 로직 변경

→ **ba-requirements → 구현 → qa-tester** 로 라우팅

### Level 3: 전체 플로우 (Full)
신규 도메인, 설계 필요, 다수 파일 수정

**트리거 키워드:**
- 새 도메인 (예: 알림, 리포트, 대시보드)
- YouTube API 새 엔드포인트 연동
- 아키텍처 변경, 배치 처리

→ **ba-requirements → system-architect → 구현 → qa-tester** 로 라우팅

### Level D: 리팩토링
코드 품질 개선, 구조 변경

**트리거 키워드:**
- 리팩토링, 정리, 개선, 최적화
- 중복 제거, 코드 품질, 테스트 커버리지

→ **code-reviewer → 구현 → qa-tester** 로 라우팅

---

## Step 3: 계획 수립

### TrendRadar 작업 유형별 자동 참조

| 작업 유형 | 자동 확인 |
|----------|----------|
| **도메인 추가** | 기존 Entity 패턴, Flyway 번호 |
| **API 개발** | 기존 Controller 패턴, ApiResponse |
| **수집 로직** | YouTube Client 패턴, Scheduler |
| **쿼리 작업** | QueryDSL 패턴, 인덱스 |
| **테스트** | 기존 테스트 패턴, Testcontainers |

---

## 실패 처리 시스템

### 실패 루프 감지

**자동 감지 조건:**
1. 동일 파일 3회 이상 수정
2. 키워드: "안돼", "왜 안되지", "이상해", "또 실패"
3. 빌드/테스트 반복 실패

### 레벨 에스컬레이션

| 현재 레벨 | 조건 | 격상 후 | 추가 투입 |
|:---------:|------|:-------:|----------|
| 0 | 3회 실패 | 1 | task-classifier |
| 1 | 트러블슈팅 후 실패 | 2 | ba-requirements |
| 2 | 트러블슈팅 후 실패 | 3 | system-architect |
| 3 | 트러블슈팅 후 실패 | STOP | 사용자 확인 |
